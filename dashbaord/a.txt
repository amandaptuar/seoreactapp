# Dashboard KPI Documentation

This document explains the logic used to calculate each metric shown on the Limitless Admin Dashboard. All metrics are derived in real-time by analyzing the `users` table directly fetched from Supabase. 

## 1. Total Users
**Logic:** A direct count of every single user registered in the database, regardless of their subscription status.
**Code Implementation:** `users.length`

## 2. Active Users
**Logic:** A count of users whose `payment_status` is explicitly set to `"paid"`. These are the users currently maintaining an active subscription.
**Code Implementation:** `users.filter((u) => u.payment_status === 'paid').length`

## 3. New Registrations
**Logic:** The number of users who have registered within the last 24 hours. It compares the user's `created_at` timestamp with the current time minus 24 hours.
**Code Implementation:** `users.filter((u) => new Date(u.created_at) >= twentyFourHoursAgo).length`

## 4. Trial Users
**Logic:** A count of users who are currently in the evaluation phase. This includes any user whose `payment_status` is set to `"trial"` or `"demo"`.
**Code Implementation:** `users.filter((u) => u.payment_status === 'trial' || u.payment_status === 'demo').length`

## 5. Monthly Revenue (MRR)
**Logic:** The current Monthly Recurring Revenue. It is calculated dynamically by taking the total number of `Active Users` (paid users) and multiplying it by the flat subscription rate of $79.
**Code Implementation:** `activeUsers * 79`

## 6. Annual Revenue (ARR)
**Logic:** The Annual Recurring Revenue projection. It is calculated by taking the Monthly Revenue (MRR) and multiplying it by 12 months.
**Code Implementation:** `MRR * 12`

## 7. Paid Conversion
**Logic:** The percentage of all registered users who have converted into actively paying subscribers. It is calculated by dividing `Active Users` by `Total Users`.
**Code Implementation:** `(activeUsers / totalUsers) * 100`

## 8. Assessments Taken
**Logic:** The total number of users who have successfully generated a cognitive assessment report. It is calculated by counting users who have a valid `report_json` attached to their profile.
**Code Implementation:** `users.filter((u) => u.report_json).length`

## 9. Monthly Growth & Revenue Chart
**Logic:** A 6-month historical line chart showing how many users registered (`created_at`) and how much MRR was generated by those specific cohorts of users. It uses `Date.getMonth()` to bucket the registrations dynamically into the last 6 calendar months.
**Code Implementation:** Aggregates `users.forEach` over a 6-month window generated dynamically from `new Date()`.

## 10. Plan Distribution Chart
**Logic:** A dynamic Donut Chart visualizing the breakdown of user statuses. It directly uses the same real-time `activeUsers` (Paid), `trialUsers` (Trial/Demo), and inactive calculation logic to render proportions.
**Code Implementation:** Renders `[ { name: "Paid ($79)", value: activeUsers }, { name: "Trial/Demo", value: trialUsers }, { name: "Inactive", value: totalUsers - activeUsers - trialUsers } ]`

## 11. Daily Registrations Chart
**Logic:** A 10-day historical bar chart showing the exact number of new users who registered on each specific day. It dynamically generates the last 10 days relative to the current date and maps user `created_at` timestamps into those daily buckets.
**Code Implementation:** Uses a sliding 10-day window `for (let i = 9; i >= 0; i--)` and compares the `dateStr` of `created_at` against the bucket.
